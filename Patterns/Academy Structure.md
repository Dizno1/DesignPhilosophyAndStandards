# Academy Structure Pattern

## Purpose

The Academy structure supports long-form instructional content that must remain usable for keyboard users, screen reader users, low vision users, mobile users, and learners with different levels of technical experience.

## When to use this pattern

Use this pattern for applications that deliver structured learning content, including courses, modules, lessons, review lessons, libraries, and study paths.

## Core structure

An Academy application should organize content by:

- Course.
- Domain or module.
- Lesson.
- Part or section.
- Review or synthesis lesson.
- Optional library reference material.
- Progress tracking.
- Practice or reinforcement activities.

## Page title standard

Page titles should begin with the specific page or lesson name and end with the application or site name.

The home page may use the course or application name as the primary title.

Lesson pages should identify the domain and lesson rather than repeating the course name as the main heading.

## Heading standard

The home page uses the course title as the H1.

A lesson page uses the domain and lesson title as the H1.

Section headings should describe the current part of the lesson.

Duplicate heading labels should be avoided unless they communicate a real structural relationship.

## Lesson structure

A standard instructional lesson may include:

- Lesson overview.
- Opening scenario.
- Core learning.
- Lesson-specific applied section.
- Real world examples.
- Professional perspective.
- Accessibility implications.
- Common misconceptions.
- Exam focus or applied review.
- Lesson summary.

The exact labels may vary when the lesson requires a different teaching flow, but the structure should remain predictable.

## Review lesson structure

A review lesson may be longer than a standard lesson and may include synthesis across lessons or domains.

Review lessons should help learners:

- Connect concepts.
- Compare related ideas.
- Apply professional judgment.
- Identify exam traps.
- Prepare for practice questions.

## Library behavior

Related library links should be relevant to the current lesson.

The library should support learning without becoming repetitive or visually cluttered.

Do not repeat generic library headings throughout the interface when more specific headings would improve navigation.

## Progress behavior

Progress should communicate useful information without overwhelming the learner.

Progress messages should distinguish between:

- Course progress.
- Domain progress.
- Lesson position.
- Lesson completion.

Screen reader announcements should be concise and should not interrupt reading unless the learner has taken an action that requires confirmation.

## Screen reader expectations

The Academy structure should support efficient navigation by:

- Headings.
- Landmarks.
- Links.
- Buttons.
- Form controls.

A learner should be able to understand where they are, what lesson they are in, what part they are reading, and what action is available next.

## Mobile and reflow expectations

Lesson content should reflow into a single column on narrow screens.

Navigation, part controls, progress information, and related resources should remain available without requiring horizontal scrolling.

## Preservation note

The Academy structure is an official Open Door Design pattern. Future learning applications should begin from this pattern rather than creating a new instructional architecture from scratch.
