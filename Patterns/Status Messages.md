# Status Messages Pattern

## Why this pattern exists

Status messages reassure users that the software understood the action.

They should reduce uncertainty, not add noise.

## Rules

- Status messages should be placed near the relevant workflow.
- Important status messages should be available to assistive technology.
- Routine status messages should use polite announcements.
- Error messages should explain what happened and how to recover.
- Do not announce results before audio or time-sensitive content has finished.

## Examples

Good:

- Practice sequence started. Result will appear after the call is heard.
- Sequence complete. Last call was Gina on the right side, regular 3.
- File converted. Download is ready.

Avoid:

- Done.
- Error.
- Something went wrong.
- Technical error codes without plain language recovery.
