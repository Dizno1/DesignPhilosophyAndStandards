# Keyboard Shortcuts Pattern

## Purpose

Keyboard shortcuts can increase speed for frequent users. They must never replace standard keyboard access.

## Core rule

Every feature must work without custom shortcuts. Shortcuts are accelerators, not the only path.

## Required behavior

- Shortcuts are documented in the application.
- Shortcuts do not conflict with common screen reader commands where practical.
- Shortcuts do not conflict with browser or operating system commands where practical.
- Users can complete the workflow with Tab, Shift+Tab, Enter, Space, arrow keys, and standard controls.
- Single-letter shortcuts are used cautiously because they may conflict with screen reader quick navigation.

## Preferred approach

Use visible controls first. Add shortcuts only for repeated actions or training environments where speed matters.

## Example shortcut help

A shortcut help section should be reachable from the page and include every shortcut in plain language.

Example:

- S: Start practice
- N: Next question
- R: Repeat question
- H: Return home

## Screen reader considerations

Many screen readers use letter keys for quick navigation. Applications should not rely on single-letter shortcuts unless the context is controlled and the user has clear instructions.

## Common mistakes

- Hiding core actions behind shortcuts.
- Using undocumented shortcuts.
- Creating shortcuts that conflict with assistive technology.
- Changing shortcut meanings between applications.
