# Open Door Design Philosophy

Open Door Design exists to remove barriers between people and technology.

Software should adapt to the user instead of requiring the user to adapt to the software. Accessibility is not a feature added after development. It is the foundation upon which every design decision is made.

Every Open Door Design application should reduce frustration, increase productivity, and allow people of all abilities to accomplish meaningful work with confidence.

## Mission

Create software that is naturally accessible, highly productive, and simple enough that users can focus on accomplishing their goals instead of learning complicated technology.

We are not building accessible versions of existing software.

We are rethinking how software should work from the beginning.

## Core philosophy

The software works for the user.

The user never works for the software.

This principle overrides every other design decision.

## Guiding principles

### Purpose before technology

Users should describe what they want to accomplish.

The software determines how to accomplish it.

Users should not be required to understand file formats, codecs, compression methods, AI models, or other technical implementation details unless they specifically request advanced options.

### Accessibility first

Accessibility is never added later.

Accessibility is the starting point.

Every design decision must consider keyboard users, screen reader users, low vision users, speech recognition users, people with cognitive disabilities, and people with limited technical experience.

### Screen reader friendly

Compatibility is not enough.

Navigation should feel natural. Announcements should provide meaningful information. Focus should remain predictable. Controls should be clearly named. Progress should be communicated without requiring visual feedback.

### Simplicity

Every workflow should require as few decisions as possible.

Every unnecessary question increases complexity.

Advanced options should exist, but remain out of the way unless explicitly requested.

### Productivity

Every feature should do at least one of these things:

- Save time.
- Reduce confusion.
- Improve accessibility.
- Increase confidence.

If a feature does none of those things, it should not be included.


## Goal-Oriented Interaction
Users describe the outcome they want. Applications determine the workflow and hide unnecessary technical complexity.
